## 2. Противоречия в системе аналитики

### 2.1. Soft Launch vs Технические ограничения

| Аспект | [09_План_разработки.md:88-90](Docs/Plan/09_План_разработки.md:88) | [10_Технические_требования.md:59](Docs/Plan/10_Технические_требования.md:59) |
|--------|----------------------------------------|----------------------------------------|
| Цель | Тестирование на реальных пользователях | Аналитика — НЕ ТРЕБУЕТСЯ |
| Метрики | Предполагается сбор данных | Запрещено |

**Проблема:** Soft Launch традиционно предполагает сбор метрик ( retention, conversion, session length), но проектный запрет на аналитику делает эту цель недостижимой.

**Рекомендация:** Переформулировать цели Soft Launch как "стабильность и багфиксинг" без метрик, либо добавить серверную инфраструктуру для аналитики.

**Проблема:** План предусматривает регулярный выпуск нового контента (уровни, события), но архитектура на ScriptableObject требует полной пересборки билда для каждого изменения.

**Конфликт:** Live Ops в классическом смысле (добавление контента без обновления приложения) невозможен.

## 5. Противоречия в технических требованиях

### 5.1. Оптимизация BFS

| Аспект | [03_Алгоритмы_и_системы.md:116](Docs/Plan/03_Алгоритмы_и_системы.md:116) | [10_Технические_требования.md:80](Docs/Plan/10_Технические_требования.md:80) |
|--------|----------------------------------------|----------------------------------------|
| Object Pooling | Рекомендуется | Рекомендуется |
| Статус | Не обязательно | Не обязательно |

**Проблема:** Оба документа рекомендуют Object Pooling, но не делают его обязательным. При этом множественные вызовы BFS при каскадах создают высокую нагрузку на GC.

**Рекомендация:** Object Pooling для BFS должен быть ОБЯЗАТЕЛЬНЫМ требованием, а не рекомендацией.

### 5.2. RAM Usage

| Аспект | [10_Технические_требования.md:42](Docs/Plan/10_Технические_требования.md:42) | [10_Технические_требования.md:67](Docs/Plan/10_Технические_требования.md:67) |
|--------|----------------------------------------|----------------------------------------|
| RAM target | < 300 MB | < 300 MB |
| GC per frame | — | < 2 KB |

**Проблема:** Требование < 2 KB GC per frame при 60 FPS создаёт ограничение в 120 KB GC budget в секунду. Это очень строгое требование для C# с каскадными анимациями.

**Влияние:** Реализация множественных каскадов с анимациями может превысить GC budget.

## 6. Противоречия в системе сложности

### 6.1. Win Rate для Hard уровней

| Сложность | [05_Прогрессия_и_сложность.md:165-168](Docs/Plan/05_Прогрессия_и_сложность.md:165) |
|-----------|----------------------------------------|
| Easy | 80-90% |
| Medium | 60-70% |
| Hard | 30-50% |
| Expert | 20-40% |

**Проблема:** Win Rate 30-50% для Hard и 20-40% для Expert может восприниматься игроками как:
- Нечестная сложность (unfair difficulty)
- Вынужденная монетизация (pay-to-win)
- Баг в балансировке

**Рекомендация:** Пересмотреть целевые Win Rates:
- Easy: 85-95%
- Medium: 70-80%
- Hard: 50-65%
- Expert: 35-50%

## 10. Рекомендации по устранению

### 10.1. Фиксация количества уровней

**Текущее состояние:** Противоречие между Roadmap и Production спецификациями.

**Решение:** Установить фиксированные значения:
- **Alpha:** 35 уровней ✓ (исправлено в 09:48)
- **Beta:** 55-60 уровней (финальная полировка)
- **Soft Launch:** 70 уровней
- **Global Launch:** 80 уровней

### 10.2. Уточнение Soft Launch

**Текущее состояние:** Конфликт между целями и техническими ограничениями.

**Решение:**
- Фокус на стабильности и багфиксинге
- Аналитика [ЗАПРЕЩЕНА]
- A/B тестирование [ТРЕБУЕТ СЕРВЕРОВ]
- Добавить примечание в документы

### 10.3. Live Ops

**Текущее состояние:** Заявлен Live Ops, но архитектура не поддерживает.

**Решение:** Выбрать один из вариантов:

**Вариант A:** Отказ от Live Ops в классическом смысле
- Все 80 уровней доступны с релиза
- Обновления только с пересборкой билда

**Вариант B:** Добавить Asset Bundles в архитектуру
- Требует изменения техтребований
- Увеличивает сложность разработки

### 10.4. Pre-production критерии

**Решение:** Добавить конкретные критерии выхода:
- Время сессии > 5 минут
- Win Rate на тестовых уровнях > 70%
- Количество перезапусков < 3
- Положительный фидбек от тестировщиков

### 10.5. Object Pooling

**Решение:** Изменить статус с "рекомендуется" на "обязательно":
- Добавить в [03_Алгоритмы_и_системы.md](Docs/Plan/03_Алгоритмы_и_системы.md) секцию об обязательности
